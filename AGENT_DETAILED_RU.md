# Инструкция для ИИ-агента (максимально подробно)

Читай сверху вниз. Делай шаги по порядку. Не пропускай шаги.

---

## 0. Главная идея (прочитай 3 раза)

Есть ДВА компьютера:

1. **Компьютер ВЛАДЕЛЬЦА** (хозяин проекта, GPU, датасет `data/`, сервер на порту 228)
2. **Компьютер ДРУГА / ТВОЙ** (где ты, агент, правишь код)

Ты **НЕ** подключаешься к диску владельца. Ты **НЕ** запускаешь `.bat` на его ПК. Ты **НЕ** имеешь доступа к его Windows.

Ты делаешь только две вещи:

1. **Git push** — отправить код в GitHub
2. **POST /api/deploy** — попросить компьютер владельца: скачать новый код и перезапустить сервер

GitHub = облако с кодом.  
Deploy URL = кнопка «обновись» на машине владельца.

## 1. Что тебе выдаст владелец (спроси в личке)

| Что | Зачем | Пример |
|-----|-------|--------|
| Ссылка на GitHub репо | клонировать код | https://github.com/LilVanya1/ript-mimii |
| PUBLIC_URL | дергать deploy | владелец пришлёт, например `https://xxxx.ngrok-free.dev` |
| DEPLOY_TOKEN | пароль для deploy | длинная строка, **только в личке**, не в git |

Без PUBLIC_URL и DEPLOY_TOKEN deploy **не сработает**.

**Важно:** PUBLIC_URL работает только если у владельца **включён** сервер (`start_public.bat` или `python app.py --public`). Если сайт не открывается — напиши владельцу: «запусти сервер, пришли рабочий URL».

## 4. Каждый раз когда правишь код (СТРОГО ПО ПОРЯДКУ)

### Шаг A — забрать свежий код

```bash
git pull origin main
```

Зачем: чтобы не конфликтовать с тем, что уже в GitHub.

### Шаг B — правки

Меняй только файлы внутри репозитория.  
Не трогай `.env` (его нет в git и не должен появиться).

### Шаг C — проверка синтаксиса

```bash
python -m compileall src app.py
```

Если ошибки — исправь, не пушь.

### Шаг D — коммит и push

```bash
git add .
git status
git commit --trailer "Co-authored-by: Cursor <cursoragent@cursor.com>" -m "fix: кратко что сделал"
git push origin main
```

Зачем push: код попадает в GitHub. **На ПК владельца код ещё НЕ обновился.**

### Шаг E — deploy на машину владельца

Подставь `PUBLIC_URL` и `DEPLOY_TOKEN` от владельца:

```bash
curl -X POST "PUBLIC_URL/api/deploy" \
  -H "Content-Type: application/json" \
  -H "X-Deploy-Token: DEPLOY_TOKEN" \
  -d "{\"pull\": true, \"restart\": true}"
```

**Пример** (подставь свои значения):

```bash
curl -X POST "https://xxxx.ngrok-free.dev/api/deploy" \
  -H "Content-Type: application/json" \
  -H "X-Deploy-Token: секрет_от_владельца" \
  -d "{\"pull\": true, \"restart\": true}"
```

Что происходит на ПК владельца:
1. `git pull` — скачивает твой коммит
2. через ~2 сек убивает старый процесс на порту 228
3. запускает `app.py` заново с новым кодом

### Шаг F — проверка

Подожди 5–10 секунд. Потом:

```bash
curl "PUBLIC_URL/api/state"
```

Должен прийти JSON со `status` (обычно `idle`).

## 5. Ответы deploy (что значит)

| Ответ | Значение | Что делать |
|-------|----------|------------|
| `"ok": true` | всё ок | готово |
| `401 Unauthorized` | неверный DEPLOY_TOKEN | спроси токен у владельца |
| `403 Deploy disabled` | у владельца нет `.env` | пусть создаст `.env` |
| `409 Busy` | идёт обучение | подожди 5–10 мин, повтори deploy |
| timeout / connection refused | сервер выключен или URL мёртв | напиши владельцу запустить `start_public.bat` |

## 6. Обучение модели (если просят)

### Вариант 1 — на машине владельца (через API)

После успешного deploy:

```bash
curl -X POST "PUBLIC_URL/api/train" \
  -H "Content-Type: application/json" \
  -d "{\"machine_type\":\"fan\",\"machine_id\":\"id_00\",\"snr\":6,\"model_backend\":\"mimii_baseline\",\"epochs\":50,\"batch_size\":512,\"learning_rate\":0.001}"
```

`model_backend`:
- `mimii_baseline` — официальный простой AE (сравнить с Hitachi baseline)
- `mobilenet` — текущая своя модель

**Правило:** один `machine_id` = одна модель. Не смешивай id_00 и id_02.

Смотри прогресс: `GET PUBLIC_URL/api/state` → поле `log`.

### Вариант 2 — у себя локально

```bash
python app.py
```

Открой http://localhost:228 — но нужна папка `data/`.

## 7. Чего НЕ делать (частые ошибки)

1. **Не пушить** `.env`, пароли, токены ngrok
2. **Не пушить** папку `data/` (огромная)
3. **Не ждать** что код сам появится у владельца после push — **всегда** делай deploy
4. **Не использовать** батники владельца — у тебя их нет и не надо
5. **Не менять** git remote на чужой репозиторий
6. Если ngrok у владельца не работает (ошибка 9040) — deploy по старому URL не дойдёт; жди **новый** PUBLIC_URL от владельца
